# Subagent observations: web-search-researcher

Date: May 14, 2026

* 🟡 WorkOS User ID example: `user_01E4ZCR3C56J083X43JQXF3JK5` — `user_` prefix + 26-char ULID = 31 chars total; max length not explicitly stated in docs but safe at 36 chars (`text` fine)
* 🟡 WorkOS Org ID example: `org_01EHZNVPK3SFK441A1RGBFSHRT` — `org_` prefix + 26-char ULID = 30 chars; same pattern
* 🟡 WorkOS Organization Membership ID: `om_` prefix + 26-char ULID; fields: id, user_id, organization_id, organization_name, status (active|inactive|pending), role {slug}, roles [], custom_attributes, directory_managed, created_at, updated_at
* 🟡 WorkOS access token confirmed claims: sub, sid, iss (https://api.workos.com/ with trailing slash, or custom domain), org_id (NOT organization_id), role, permissions, exp, iat. No aud claim documented. Email NOT in access token.
* 🔴 WorkOS custom role slugs must begin with `org-` (hyphen, NOT colon). They appear in JWT role claim as-is (e.g. `org-billing-admin`). Built-in roles use base slugs (member, admin).
* 🟡 WorkOS act claim shape for impersonation: object with sub field containing impersonator email — `{"act": {"sub": "admin@foocorp.com"}}`. Authenticate API response also returns separate impersonator object with email + reason string.
* 🟡 JWKS endpoint: `https://api.workos.com/sso/jwks/<clientId>`
* 🟡 WorkOS webhook events: user.created/updated/deleted, organization.created/updated/deleted, organization_membership.created/updated/deleted. Role changes fire organization_membership.updated. Envelope: {event, id, data, created_at, context}.
* 🟡 SSO-provisioned users: connection_id/connection_type NOT on User object — they live on separate Identity objects. Identity has idp_id (upstream IdP user ID) + type (OAuth) + provider. Custom SSO attributes land on organization_membership.custom_attributes, not on User.
* 🟢 WorkOS does not expose a dedicated `invitation` status — membership statuses are active/inactive/pending only (no 'invited' or 'suspended' as separate states)

Date: May 15, 2026

* 🔴 TravelClick iHotelier REST API full reference is LOGIN-GATED at developer.travelclick.com — "Please sign-in to review and test drive our REST APIs." Only API family names visible publicly (Book, Shop, Codes, Entity, Loyalty REST APIs).
* 🟡 Primary public doc artifacts: App Notes PDFs at travelclick-websolutions.com/uploads/…/app-notes-reservation-booking-v1-1.pdf and app-notes-group-reservation-v1.pdf — both are binary-encoded (compressed streams), not OCR'd; WebFetch cannot extract text. Google index has partial natural-language summaries.
* 🟡 From search index summaries of App Notes PDF: hold-reservation TTL is 10 min; after expiry "allocated room inventory is released and any subsequent reservation complete API call will fail"; client must call hold-reservation again to get new confirmation ID; expired confirmation IDs explicitly stated as invalid.
* 🔴 Q1 partial answer confirmed via search index: expired-hold confirm fails with a hard error (call "will fail"); specific HTTP status code and error body NOT documented publicly. Q2 (mid-call expiry race) and Q3 (idempotency key) have NO public documentation — not mentioned in any source found.
* 🟡 Endpoints confirmed: POST /book/v1/hotel/{hotelCode}/hold-reservation and POST /book/v1/hotel/{hotelCode}/reservation (Book REST API v1). Staging Swagger at api-staging.travelclick.com also requires auth.
* 🔴 Claude Code .claude/ gitignore convention: Anthropic explicitly documents only `settings.local.json` and `CLAUDE.local.md` as gitignore-required. Auto memory lives at `~/.claude/projects/<project>/memory/` (global, NOT in project .claude/). Runtime/session folders (`projects/`, `file-history/`, `debug/`, `todos/`, `tasks/`, `session-env/`, `shell-snapshots/`, `paste-cache/`, `worktrees/`, `chrome/`, `history.jsonl`) exist in the GLOBAL `~/.claude/` dir, NOT inside project `.claude/`; so they are not a project-level gitignore concern. The only commonly runtime file created INSIDE project `.claude/` is `scheduled_tasks.lock`.
* 🟡 Claude Code docs redirected: docs.anthropic.com/en/docs/claude-code/* now 301s to code.claude.com/docs/en/*. Use code.claude.com directly.
* 🔴 "APC" in Zed context = Agent Client Protocol (ACP) — Zed's open standard (Apache license) for connecting external CLI agents (Claude Code, Gemini CLI, Codex, etc.) to the editor UI. NOT "agent panel" or anything Anthropic-coined.
* 🟡 Zed runs Claude Code via ACP adapter (@zed-industries/claude-agent-acp npm package) which wraps the Claude Code SDK and translates to ACP JSON-RPC. Claude Code runs as a background process; auth is separate (API key or Claude Pro/Max subscription, NOT Zed's own auth).
* 🔴 Zed ACP external agent gaps vs native (CONFIRMED from zed.dev/docs/ai/external-agents + GitHub issues, May 2026): (1) No editing past messages, (2) no thread history resumption, (3) no checkpointing, (4) agent teams not supported, (5) hooks not supported, (6) AskUserQuestion tool hard-disabled in adapter code, (7) several built-in slash commands (/usage /status /model /memory /permissions /agents /mcp) advertised but emit no output over ACP, (8) subagents dispatched via ACP do NOT inherit CLAUDE.md/AGENTS.md context (workaround: prepend "Read CLAUDE.md first." to subagent prompt), (9) /compact and /effort CLI commands inaccessible from Zed Agent Panel input, (10) permissions.defaultMode: "auto" has no effect — prompts as frequently as "default", (11) terminal output not streamed progressively (arrives in one chunk at tool completion), (12) skills NOT yet discoverable via slash commands in Agent Panel (PR #652 still unmerged as of 2026-05-15), (13) MCP OAuth remote servers may have issues
* 🔴 Plan mode in Zed ACP: Mr. Cobb confirmed plan mode DOES work. Prior observation that it was unsupported was from the Sept 2025 launch blog and is now stale — now supported.
* 🟡 Zed ACP billing change: starting June 15, 2026, Claude SDK (and thus ACP/Zed) no longer counts against Pro/Max subscription — moves to API billing. Opt-in credit available via email before that date. Source: github.com/zed-industries/claude-agent-acp/issues/658
* 🟡 Context window fix: Zed issue #51648 (ACP agents compacting at 200K instead of 1M) was CLOSED as fixed via PR #51695 (merged 2026-03-16). Context window now correctly uses 1M for Opus 4.6 / Sonnet 4.6.
* 🟡 Zed ACP package was renamed: @zed-industries/claude-code-acp is deprecated; new package is @agentclientprotocol/claude-agent-acp. Repo also moved/mirrored at github.com/agentclientprotocol/claude-agent-acp.
* 🟡 CLAUDE.md: root agent (primary session) DOES auto-load CLAUDE.md. Subagents dispatched via ACP do NOT (issue #623, open as of 2026-05-15). Zed does NOT forward its own rules/tool-permissions settings to external agents — Claude Agent reads its own config.
* 🟡 Claude Code surfaces as of May 2026: Terminal CLI, VS Code extension, JetBrains plugin, Desktop app, Web (claude.ai/code), iOS app. Zed integration is via ACP (beta). All surfaces share CLAUDE.md, settings, MCP servers.
